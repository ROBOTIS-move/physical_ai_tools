# ROSbag to LeRobot 데이터 동기화 메커니즘

이 문서는 ROSbag 데이터를 LeRobot 데이터셋으로 변환할 때 발생하는 **데이터 동기화 문제**와 해결 방법을 설명합니다.

## 참조 코드
- `physical_ai_server/physical_ai_server/data_processing/rosbag_to_lerobot_converter.py`

---

## 1. 문제 정의: Action이 0부터 시작하는 현상

### 1.1 ROSbag 토픽 구조

로봇 데이터 녹화 시 다양한 토픽들이 **서로 다른 시간에** 시작됩니다:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    ROSbag 토픽 타이밍 (실제 데이터)                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  토픽 유형        토픽 이름              첫 메시지    메시지 수     주파수     │
│  ─────────────────────────────────────────────────────────────────────────  │
│  STATE           /joint_states          0.000s      1908        ~100Hz     │
│  ─────────────────────────────────────────────────────────────────────────  │
│  ACTION          left_arm               0.001s      1905        ~100Hz     │
│  ACTION          right_arm              0.001s      1905        ~100Hz     │
│  ACTION          head                   0.001s      1906        ~100Hz     │
│  ACTION          lift                   0.001s      1906        ~100Hz     │
│  ─────────────────────────────────────────────────────────────────────────  │
│  IMAGE           cam_wrist_left         0.006s       572        ~30Hz      │
│  IMAGE           cam_wrist_right        0.007s       573        ~30Hz      │
│  IMAGE           cam_head               0.028s       570        ~30Hz      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.2 시간축 상의 토픽 도착 순서

```
시간(초)   0.000   0.001   0.006   0.007           0.028
           │       │       │       │               │
           ▼       ▼       ▼       ▼               ▼
    ━━━━━━━┿━━━━━━━┿━━━━━━━┿━━━━━━━┿━━━━━━━━━━━━━━━┿━━━━━━━━━━━━━━▶ t

           │       │       │       │               │
           │       │       │       │               └── cam_head 시작
           │       │       │       │
           │       │       │       └── cam_wrist_right 시작
           │       │       │
           │       │       └── cam_wrist_left 시작
           │       │
           │       └── ACTION 토픽들 시작 (left_arm, right_arm, head, lift)

    │     - State는 있지만 Action 데이터가 없음                         │
    │     - 이 구간의 프레임은 Action=0으로 채워짐                       │
    └──────────────────────────────────────────────────────────────────┘
```

## 2. 해결 방법

두 가지 핵심 솔루션을 적용합니다:

### 2.1 Solution 1: effective_min_time 조정

**위치**: `_resample_to_fps()` 메서드 (600-634줄)

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    Solution 1: 유효 시작 시간 조정                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  원본 시간축:                                                               │
│                                                                             │
│     min_time (State 시작)              max_time (State 끝)                  │
│         │                                  │                                │
│         ▼                                  ▼                                │
│    ━━━━━┿━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┿━━━━━━━━━▶ t                   │
│         │                                                                   │
│         │   first_action_time                                               │
│         │       │                                                           │
│         │       ▼                                                           │
│    ━━━━━┿━━━━━━━┿━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━▶ t                   │
│         │       │                                                           │
│         │◄─────►│                                                           │
│         │ 무효  │                                                           │
│         │ 구간  │                                                           │
│                                                                             │
│  ─────────────────────────────────────────────────────────────────────────  │
│                                                                             │
│  조정된 시간축:                                                             │
│                                                                             │
│            effective_min_time = max(min_time, first_action_time)            │
│                   │                                                         │
│                   ▼                                                         │
│    ━━━━━━━━━━━━━━━┿━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━▶ t                   │
│                   │◄───────────────────────────────►│                       │
│                   │      유효한 데이터 구간          │                       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**코드 동작**:
```python
# _resample_to_fps() 내부
effective_min_time = min_time  # State 시작 시간 (0.000s)

if action_messages:
    action_times = [t for t, _ in action_messages]
    first_action_time = min(action_times)  # 0.001s
    # State와 Action 중 늦은 시간부터 시작
    effective_min_time = max(min_time, first_action_time)
```

## 3. Causal Sync (인과적 동기화)

### 3.1 원리

Causal Sync는 **미래 데이터를 사용하지 않고**, 오직 **이전(또는 현재) 시점의 데이터**만 사용합니다.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        Causal Sync vs Nearest Sync                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  메시지 타임스탬프:    m1      m2      m3      m4      m5                   │
│                        │       │       │       │       │                    │
│                        ▼       ▼       ▼       ▼       ▼                    │
│  ━━━━━━━━━━━━━━━━━━━━━┿━━━━━━━┿━━━━━━━┿━━━━━━━┿━━━━━━━┿━━━━━━━━━▶ t       │
│                      0.10    0.12    0.14    0.16    0.18                   │
│                                                                             │
│  대상 시간(target):                   │                                     │
│                                       ▼                                     │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┿━━━━━━━━━━━━━━━━━━━━━━━━━▶ t       │
│                                      0.15                                   │
│                                                                             │
│  ─────────────────────────────────────────────────────────────────────────  │
│                                                                             │
│  ❌ Nearest Sync (사용하지 않음):                                           │
│                                                                             │
│     |0.15 - 0.14| = 0.01  vs  |0.15 - 0.16| = 0.01                         │
│     → m3 또는 m4 선택 (미래 데이터 m4 사용 가능)                            │
│                                                                             │
│  ─────────────────────────────────────────────────────────────────────────  │
│                                                                             │
│  ✓ Causal Sync (사용):                                                     │
│                                                                             │
│     target_time=0.15 이전의 메시지 중 가장 최근 것 선택                     │
│     → m3 (0.14s) 선택                                                       │
│     → staleness = 0.15 - 0.14 = 0.01s = 10ms                               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.2 구현: _find_previous_value_in_list()

**위치**: 566-598줄

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    _find_previous_value_in_list() 동작                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  입력:                                                                      │
│    - messages: [(t1, val1), (t2, val2), ...]                               │
│    - target_time: 0.15                                                      │
│    - tolerance: 0.05 (50ms)                                                 │
│                                                                             │
│  처리:                                                                      │
│    for msg_time, value in messages:                                         │
│        if msg_time <= target_time:        # ← 핵심: 이전 데이터만           │
│            if best_time is None or msg_time > best_time:                    │
│                best_time = msg_time                                         │
│                best_value = value                                           │
│                                                                             │
│  출력:                                                                      │
│    - (best_value, staleness_ms)                                            │
│    - staleness_ms = (target_time - best_time) × 1000                       │
│                                                                             │
│  ─────────────────────────────────────────────────────────────────────────  │
│                                                                             │
│  예시:                                                                      │
│                                                                             │
│    messages = [(0.10, v1), (0.12, v2), (0.14, v3), (0.16, v4)]              │
│    target_time = 0.15                                                       │
│                                                                             │
│    반복 1: 0.10 <= 0.15 ✓ → best_time=0.10, best_value=v1                  │
│    반복 2: 0.12 <= 0.15 ✓, 0.12 > 0.10 ✓ → best_time=0.12, best_value=v2  │
│    반복 3: 0.14 <= 0.15 ✓, 0.14 > 0.12 ✓ → best_time=0.14, best_value=v3  │
│    반복 4: 0.16 <= 0.15 ✗ → 건너뜀                                         │
│                                                                             │
│    결과: (v3, 10.0ms)  # staleness = (0.15 - 0.14) × 1000                  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 6. 요약

### 6.1 문제와 해결책

| 문제 | 원인 | 해결책 |
|------|------|--------|
| Action이 0으로 시작 | State가 Action보다 먼저 도착 | `effective_min_time` 조정 |
| 불완전한 Action | 4개 토픽 중 일부만 도착 | `all_topics_have_data` 체크 |
| 미래 데이터 참조 | Nearest Sync 사용 | Causal Sync (이전 값만) |

### 6.2 핵심 메서드

| 메서드 | 역할 |
|--------|------|
| `_merge_action_messages()` | 4개 Action 토픽 병합, 불완전 프레임 제외 |
| `_resample_to_fps()` | 유효 시작 시간 조정, 30fps 리샘플링 |
| `_find_previous_value_in_list()` | Causal Sync 구현 (이전 값만 탐색) |

### 6.3 결과

- **입력**: 1908 State, 7622 Action (4토픽), 1715 Image (3카메라)
- **출력**: 572 프레임 @ 30fps
- **보장**: 모든 프레임에 유효한 State와 Action 존재

## 7. 이미지 동기화 (ROSbag Recording 단계)

### 7.1 문제: 카메라별 프레임 수 불일치

기존 방식에서는 각 카메라가 독립적으로 녹화되어 프레임 수가 달라질 수 있습니다:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     기존 방식: 독립적 녹화                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  cam_wrist_left:  ──●──●──●──●──●──●──●──●──●──●──●──●──▶  572 프레임      │
│  cam_wrist_right: ──●──●──●──●──●──●──●──●──●──●──●──●──●▶  573 프레임     │
│  cam_head:        ──●──●──●──●──●──●──●──●──●──●──▶         570 프레임      │
│                                                                             │
│  문제:                                                                      │
│    - 녹화 시작/종료 시점 차이                                                │
│    - 네트워크 지연으로 인한 메시지 드랍                                       │
│    - LeRobot 변환 시 프레임 불일치                                           │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 7.2 해결책: SyncedImageCompressor

`SyncedImageCompressor`는 ROSbag 녹화 단계에서 Causal Sync를 적용합니다:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     새로운 방식: 동기화된 녹화                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 초기화 단계 (150ms)                                                     │
│     - 모든 카메라에서 첫 프레임 수신 대기                                     │
│     - 가장 가까운 timestamp 조합 찾기                                        │
│     - anchor_timestamp 설정 (가장 늦은 timestamp)                           │
│                                                                             │
│  2. 고정 주기 샘플링 (30fps)                                                │
│     - target_time = anchor + frame_idx × (1/30)                            │
│     - 각 카메라에서 target_time 이전의 최신 프레임 선택 (Causal Sync)        │
│     - 모든 카메라 프레임을 동시에 RAM 버퍼에 저장                            │
│                                                                             │
│  3. 청크 단위 백그라운드 인코딩 (30초)                                       │
│     - 취득 중 CPU 부하 최소화                                               │
│     - 30초마다 백그라운드 스레드에서 MP4 인코딩                              │
│     - 최종적으로 청크들을 concat하여 단일 MP4 생성                           │
│                                                                             │
│  4. 종료 시 프레임 수 검증                                                  │
│     - 모든 카메라 프레임 수 일치 확인                                        │
│     - sync_stats_report.json 생성                                          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```
